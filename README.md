# meili-indexer

Наповнює пошук Meilisearch для lartek.com.ua з YML-фіду Horoshop.
Запускається на Railway за розкладом (`railway.json` → cron, раз на 6 год).

## Як працює (безпечне оновлення)

1. Качає фід → **гард #1**: перевіряє, що це XML з `<offer>` (а не анти-бот
   заглушка / HTML / порожньо).
2. Мапить у документи (нормалізація розмірів: `30*52*10` = `30×52×10` = …).
3. **Гард #2**: кількість товарів адекватна (≥ `MIN_DOCS`) і без різкого
   падіння (≤ `MAX_DROP_RATIO`) — інакше оновлення скасовується, живий пошук
   не чіпається.
4. Будує **тимчасовий** індекс `products_build`, заливає туди, перевіряє
   повноту.
5. **Атомарна підміна** (`swap-indexes`): `products` миттєво отримує свіжі дані;
   пошук ніколи не бачить напів-оновлений стан. Старі/зняті товари зникають
   автоматично (щоразу свіжий індекс).

**Fail-safe:** за будь-якої помилки живий індекс лишається старим (стале, але
не зламане). Зламати пошук битий фід не може.

## Змінні (Railway → Variables)

| Змінна | Призначення | Типово |
|---|---|---|
| `MEILI_KEY` | майстер-ключ Meilisearch (обов'язково) | — |
| `MEILI_HOST` | адреса Meilisearch | вписано в коді |
| `FEED_URL` | URL YML-фіду | вписано в коді |
| `MEILI_INDEX` | назва індексу | `products` |
| `MIN_DOCS` | мінімум товарів у фіді (нижче — стоп) | `1500` |
| `MAX_DROP_RATIO` | макс. допустиме падіння кількості | `0.4` (40%) |

## Розклад

`railway.json` → `deploy.cronSchedule: "0 */6 * * *"` (раз на 6 год).
Змінити частоту — правити це значення.

## Запуск локально

```
MEILI_KEY=<master> node indexer.js
# або з іншим фідом:  node indexer.js https://.../feed.xml
```
